Hard restrictions:
- Do not emit imports.
- Do not emit Python functions.
- Do not emit classes.
- Do not emit loops.
- Do not emit if/elif/else statements.
- Do not emit executable runtime code.
- Do not call arbitrary helper functions.
- Do not model concrete file paths or raw file I/O.
- For standalone variables declared with var("name", Type), references in predicates must use Var("name").
- For standalone variable updates, use Set("name", value). Do not write Set(Var("name"), value).

Modeling rules:
- Declare workflow state using entity(...) records or var(...).
- Initialize every declared field and variable in init(...).
- Convert workflow steps into action(...) declarations.
- Put current-state guards in requires=[...].
- Put externally visible effects in emits=[...] only when a property needs to reason about that effect.
- Use Emitted("event_name") or SeenEvent("event_name") only for events that appear in some action emits=[...].
- Do not emit one event for every ordinary state transition.
- Put state updates in changes=[...].
- Model uncertain external outcomes as Bool or Enum state fields and separate guarded actions.
- Include properties for completion, retries, fallback, invariants, or success reachability when the markdown states them.
- Use invariant(...) for state predicates that must always hold.
- Use completion_requires(...) only for same-state completion safety: when outcome is true, condition must already be true in that same state.
- Use obligation(...) for eventual progress or liveness requirements. Do not encode "eventually succeeds", "eventually ships", or "eventually handles" requirements as completion_requires(...).
- Use forbidden(...) for explicitly invalid states or transition patterns.
- If the markdown describes a missing branch or unreachable success, preserve that bug in the DSL instead of inventing the missing action.
- If behavior is ambiguous, model nondeterminism explicitly rather than assuming constants.
- Use init(...) only for valid starting states, not to enumerate all workflow states.
- Do not encode uncertain future outcomes as fixed booleans in init(...).
- Use AnyOf(Bool) (or AnyOf(Enum(...))) in init(...) when a field represents an uncertain external outcome; this lets TLC explore both branches instead of freezing one.
- Any Bool field that appears in an action's requires guard MUST be initialized with AnyOf(Bool) in init, unless it exclusively represents a process-tracking flag that is always False at workflow start (e.g. a "completed" or "submitted" flag that only becomes True through an action). Properties of the input (e.g. is_alcohol, has_itemization, exceeds_limit, user_approved) are uncertain and must use AnyOf(Bool).
- Never declare integer fields with bare Int when the spec implies a finite range. Use IntRange(lo, hi) instead, deriving the bounds from the spec (e.g. "up to 3 retries" → IntRange(0, 3), "between 1 and 5 items" → IntRange(1, 5)). This keeps TLC state spaces finite.
- Prefer AnyOf(IntSet(...)) over AnyOf(IntRange(...)) for fields with policy thresholds. Pick one representative value below, at, and just above each threshold. For example, if the spec checks amount > 50 and amount > 2500, use AnyOf(IntSet(0, 49, 50, 51, 2499, 2500, 2501)) instead of AnyOf(IntRange(0, 2500)). This avoids enumerating thousands of states.
- Use AnyOf(IntRange(lo, hi)) only when the field has no thresholds and needs exhaustive range coverage (e.g. a small retry counter).
- Do NOT create a Bool field that is logically derived from another field already in the model. Use the predicate directly in action guards instead. For example, if amount=IntSet(...) is already declared, do not add exceeds_limit=Bool — instead use Gt(Field("expense", "amount"), 2500) directly in requires=[...]. Redundant derived flags create contradictory init states when both are AnyOf.
- CRITICAL: AnyOf(...) is ONLY valid inside init(...) as the value of an Eq(...) assignment. It is NOT a type and must never appear inside entity(...) or var(...). The field type and the init value are separate: entity declares the type (IntSet, IntRange, Bool, Enum), init uses AnyOf to make that value non-deterministic.
  - Correct: entity("e", Record(amount=IntSet(0, 50, 51))) and init(Eq(Field("e", "amount"), AnyOf(IntSet(0, 50, 51))))
  - Wrong: entity("e", Record(amount=AnyOf(IntSet(0, 50, 51))))
- If a step may succeed or fail, model separate actions or explicit branching in actions.
- Do not guard creation/start actions on fields of an entity whose existence is false.
- Prefer canonical start states over placeholder record values that contradict existence flags.
- Correct examples:
  - Predicate reference: Var("order_exists")
  - Predicate equality: Eq(Var("order_exists"), False)
  - State update: Set("order_exists", True)
  - Forbidden property: forbidden("cannot_cancel_paid_order", when=And(...))
- Incorrect examples:
  - Eq("order_exists", False)
  - Not("order_exists")
  - Set(Var("order_exists"), True)
  - forbidden("cannot_cancel_paid_order", And(...))

Exploration-safe examples:
```python
# Bad: freezes one branch — TLC can only explore the False path
var("payment_succeeds", Bool)
init(
    Eq(Var("payment_succeeds"), False),
)
action("payment_attempt_succeeds", requires=[Var("payment_succeeds")], changes=[...])
action("payment_attempt_fails", requires=[Not(Var("payment_succeeds"))], changes=[...])
```

```python
# Better: no bool field needed — two actions are naturally non-deterministic
action("payment_attempt_succeeds", requires=[...], changes=[...])
action("payment_attempt_fails", requires=[...], changes=[...])
```

```python
# Also valid: keep the bool field but initialize it non-deterministically
entity("payment", Record(
    status=Enum("pending", "succeeded", "failed"),
    attempt_succeeds=Bool,
))
init(
    Eq(Field("payment", "status"), "pending"),
    Eq(Field("payment", "attempt_succeeds"), AnyOf(Bool)),  # TLC explores both TRUE and FALSE
)
action("payment_attempt_succeeds", requires=[Eq(Field("payment", "status"), "pending"), Field("payment", "attempt_succeeds")], changes=[...])
action("payment_attempt_fails", requires=[Eq(Field("payment", "status"), "pending"), Not(Field("payment", "attempt_succeeds"))], changes=[...])
```

```python
# Bad: amount pinned to 0 — TLC never explores branches that depend on amount thresholds
```

```python

init(
    Eq(Field("expense", "status"), "draft"),

)
```
